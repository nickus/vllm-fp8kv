## BLOCKER findings

### B1. UPSTREAM.md Contribution #1 (the "dtype-blind autotune key" bug report) is technically false and must not be filed

- **The mechanism doesn't exist on the claimed stack.** Triton 3.7.1's `Autotuner.run` appends `str(arg.dtype)` of *every* tensor argument to the autotune cache key (`triton/runtime/autotuner.py:218-222`, verified in a local 3.7.1 install — the version RESULTS.md itself pins). The fp8 path passes `k_buffer` as `uint8`, bf16 as `bfloat16` → the cache keys already differ; fp8 can never "silently reuse the config Triton cached for bf16". The only mode where the key is dtype-blind is vLLM's opt-in `VLLM_TRITON_FORCE_FIRST_CONFIG` override (`vllm/triton_utils/force_first_config.py:40`), default off; no script, log, or note anywhere in either project sets it.
- **The claimed fix cannot produce the claimed number.** Upstream's config lists are `_FINAL_AUTOTUNE_CONFIGS`: BLOCK_N=16 only, 2/4 warps; `_SPLIT_AUTOTUNE_CONFIGS`: BLOCK_N=32 only, 4 warps (`triton_mla_sparse_kernel.py:31-38`). "Its own autotuned config (BLOCK_N=64, 8 warps)" is **not in the config space** — no key change can select it. The real (defensible) finding is that upstream's config *lists* are bf16-oriented and an fp8 path needs its own, larger configs.
- **The 608→201 µs table is a cross-kernel comparison.** 608 µs is the patched upstream kernel (single-pass final kernel at bs=32 — dispatch math: `32 × 4 ≥ 82` SMs → `num_kv_splits=1`, BLOCK_N=16). 201 µs is from `cfg.py`, a *different scratch kernel* with a broken split reduction (see B2), at a config upstream doesn't have. The "one-line fix" was never applied to the real kernel and re-measured.
- **Impact:** filing this gets refuted by the first Triton-literate reviewer. **Correction:** re-frame as "config lists need per-dtype entries", and only after measuring the actual patched kernel with extended configs.

### B2. The headline parity numbers (0.85–1.02×) and the 0.999873 correctness figure were measured on a mathematically invalid kernel

`final.py` and `cfg.py` (scratchpad) implement split-KV as: each split computes a **locally normalized** softmax mean `o_sp = Σ_j exp(qk_j − m_sp)·v_j / Σ_j exp(qk_j − m_sp)`, then combines splits with `tl.atomic_add(O, o)` (`final.py:50-51`, `cfg.py:43-44`) — a plain **unweighted sum of per-split means**. The correct global softmax is the LSE-weighted convex combination `Σ_sp w_sp·o_sp` with `w_sp = Σ_{j∈sp} e^{s_j} / Σ_all e^{s_j}` (upstream does exactly this with a partial buffer + merge kernel). With SPLITS=8 the output is ~8× too large and directionally wrong whenever attention is peaked. It survived its own check because:
- cosine is **scale-invariant** (the ~8× error is invisible), and
- i.i.d. Gaussian test data makes all per-split means nearly parallel — the degradation from 0.999996 to **0.999873** is the *signature of the broken merge*, misread as quantization noise;
- `final.py:99-106` checks only cosine + finiteness — it dropped the `max_abs < 5e-2` gate that the repo's own harness (`verify/run_all.py:96`) uses, which would have failed instantly.

**Consequences:** (a) the AC3 "apples-to-apples" table and the headline "1.756× KV capacity at ~0.85–1.0× decode speed" are timings of a kernel that does not compute attention and omits the dtype-independent merge pass; (b) 0.999873 is meaningless and must be struck; (c) the only numbers ever measured on the *real* patched upstream kernel are **0.92× / 0.45×** (commit `12a6a75`, via `test_patched.py`). The honest current state of the shipped artifact is 0.45–0.92× of bf16, with an *unverified hypothesis* that per-dtype configs recover parity.

### B3. The checked-in patch is stale relative to its generator, and the generator's split-kernel fix is a silent no-op

- `patches/triton_mla_sparse_fp8.patch` (224 lines, last regenerated at `12a6a75`) **lacks the autotune-key hunk** added to `make_fp8_kernel_patch.py` at `83dd0fa` (that commit touched only RESULTS.md, the generator, and a committed `.pyc`). Verified by regenerating against pr47629: 236-line patch containing the `@triton.autotune` replacement vs. the checked-in one without it.
- Even regenerated, the generator replaces the literal `key=["index_topk", "kv_group_num"],` — but upstream's split kernel key is `["index_topk", "NUM_KV_SPLITS", "kv_group_num"]`. The replace silently matches nothing, so the **split kernel — the path actually taken at decode bs≤32 — never gets the key change**, directly contradicting the generator's own comment. No error is raised (this replace is outside the anchor-checked loop).
- What does hold: both patch variants `git apply --check` cleanly on `bbe2ab4d6` and `py_compile` afterward; all six mandatory anchors are present; the `_thread_is_fp8` multi-replaces hit exactly the intended 3 signatures / 2 call sites / 2 launches.

### B4. AC1 is marked ✅ but no vLLM engine ever ran with fp8 KV — and the shipped code cannot boot one

- `TritonMLASparseBackend.get_kv_cache_shape` returns `(num_blocks, block_size, head_size=576)` for **every** cache dtype; `fp8_ds_mla` needs 656. `backend_patch.py` doesn't patch it, so a real boot would allocate a wrong-shaped cache before any kernel runs.
- `_canonicalize_sparse_mla_kv_cache_dtype` maps `fp8`/`fp8_e4m3` → `fp8_ds_mla` only for FLASHMLA_SPARSE and SM120 — not for TRITON_MLA_SPARSE. AC1's literal flag `--kv-cache-dtype fp8_e4m3` is neither mapped nor declared.
- No `LLM()`/serve invocation exists anywhere in the project except `capture_indices.py`, which was never successfully run (see M8) and defaults to `kv_cache_dtype="auto"` anyway.
- RESULTS.md "AC1 fp8 boots + parity ✅ correctness proven at every layer" silently substitutes kernel-level cosine on synthetic Gaussian data for engine boot + logits parity. No logits were ever compared. This is the clearest case of AC reinterpretation in the repo.

## MAJOR findings

**M1. `verify/integration.py` does not do what its name or docstring says.** Docstring: "Compares fp8-KV output against the SAME backend running bf16-KV… that is the AC1 parity gate." The code compares against the repo's **own fp32 reference** (`integration.py:94`), constructs no attention metadata, and never calls `forward_mqa`, `_forward_bf16_kv`, or `_forward_fp8_kv` — it only `hasattr`-checks the monkeypatch. Commit `dfd518b` "INTEGRATION GREEN: fp8-KV works inside vLLM's real TRITON_MLA_SPARSE backend" — the backend's forward path was never executed.

**M2. Nothing shipped verifies the patched upstream kernel.** The 0.999996 "inside upstream's real split-KV kernel" result came from `test_patched.py`, which lives only in the (ephemeral) scratchpad and imports from `/root/vllm-fp8kv`. RESULTS.md also conflates it with the index-converter test (that one ran with the *standalone* kernel). Neither `verify/run_all.py`, `contract.py`, nor `integration.py` ever imports `triton_mla_sparse_attention` for correctness; `bench.py` uses it for bf16 *timing* only.

**M3. Latent crash in shipped `backend_patch._forward_fp8_kv`**: `num_tokens = q.shape[0]` executes **before** `if isinstance(q, tuple): q = torch.cat(...)` — a tuple `q` (the documented input form) raises `AttributeError`. Unreachable via `_forward_mqa_fp8_aware` (which cats first), but it's the exact kind of dead-but-broken branch that proves the function is untested: no test executes either function (M1).

**M4. The upstream patch is inert standalone.** It touches only the kernel file; upstream's backend still raises on quantized KV, still declares bf16-only dtypes, and still allocates 576-wide bf16 caches — so `is_fp8` can never become true from a real engine. The backend half exists only as the out-of-tree monkeypatch, which is itself engine-incomplete (B4). UPSTREAM.md's "one flag, one branch, one dispatcher arg" materially undersells a mergeable feature.

**M5. AC2 spin.** 1.756× vs the ≥1.9× target is a **miss reframed as PASS** (the ceiling argument `1152/656` is arithmetically correct and worth stating, but the AC failed as written). It was also never *measured* — `bench.py:64-72` prints byte arithmetic; README says "Verified on RTX 3090 … 1.756× KV pool". And at engine level the fp8 indexer K-cache (~132 B/token, identical in both configs) dilutes the real pool gain to ~1.63×.

**M6. AC3's axis was quietly swapped.** The AC specified context-length gates (≥95% @8K, ≥100% @64K). `final.py` measures a batch-size sweep at one fixed pool (131072 slots) and fixed topk=2048. No 8K-vs-64K comparison exists for any kernel.

**M7. The "drift canary" partially checks itself.** `contract.py:63-64` computes `scale_byte=(512//4)*4` and `rope_byte=(512//2+8)*2` locally — pure arithmetic on the constant 512, compared against the repo's own constants — despite the file's claim that "none of our constants are trusted here". Only row-bytes is genuinely derived from vLLM. The k_scale=2.0 tripwire and writer-roundtrip, by contrast, are real.

**M8. The trace-replay was never run, and as shipped can't work on the project's own hardware.** `contract.py::check_indices_contract` skips without `/root/idx_trace.pt`. `capture_indices.py:33-40` hooks the Impl class in `flashmla_sparse` — a backend that is *not selected* on RTX 3090 (TRITON_MLA_SPARSE is), so the hook would never fire. Commit `d5df3c1`'s "live index traces" describes a capability, not an executed test.

Reasonable claim, wrong verb. Also: upstream's *own* PR already ships uint8-LUT software fp8 decode on SM80 (`mqa_logits_triton.py:191`) — an in-PR precedent that would strengthen the RFC but also weakens the novelty framing; not cited.
- **m6. Cosmetic:** misindented `IS_FP8=is_fp8,` in the final-kernel launch hunk (compiles fine); `cfg.py`'s sweep has zero correctness gate (`try/except: pass`) — the methodology that crowned the "winning config".

## What is genuinely solid (credit where due)

- The **bit-math fp8e4m3fn decode is correct** — re-derived the algebra (payload lands uniformly 2⁻⁸ off in fp16 for normals *and* denormals; NaN select matches torch CPU semantics), and the 256-value bit-exactness tests, including the CUDA-canonical-NaN caveat, are well designed.
- The **standalone kernel** (`vllm_fp8kv/fp8_ds_mla_sparse_decode.py`) is mathematically sound: single-pass online softmax (no split-merge to get wrong), correct per-tile scale broadcast, raw-bf16 RoPE, int64 slot offsets with a real 3.5M-slot boundary test, and a load-bearing NaN guard for leading-masked chunks.
- The **k_scale=2.0 tripwire** against vLLM's own C++ writer is an excellent contract test; the layout/scale/ENABLE_FP8 claims all corroborate against upstream source.
- Both patch variants apply cleanly to pr47629 head and py_compile; anchors are robustly checked (in the main loop).
- The author self-retracted the 1.88× claim and openly listed AC4/AC6/AC8 as not reached.

## Verdict on UPSTREAM.md (as it stood at `d33358f`)

- **Contribution #1 (autotune bug report): DO NOT FILE.** Central mechanism false on Triton ≥3.x, the claimed winning config isn't in upstream's config space, and the 3.0× number compares two different kernels — one of which doesn't compute attention.
- **Contribution #2 (feature RFC): file only after rework.** Safe as-is: the gap quotes, the XPU precedent, the fp8-as-storage vs native-fp8 distinction, the scale-contract/RoPE/int64 reviewer notes, and the 0.999996 writer-roundtrip correctness line. **Not safe:** the 0.999873 lower bound, the "0.85–1.02× decode speed" line, "non-Ampere regression: none", the "prerequisite" reference to contribution #1, and the patch itself (stale, split-key no-op, engine-inert without a backend diff including `get_kv_cache_shape` and dtype canonicalization).
- **Minimal path to fileable:** regenerate the patch; fix or drop the key story; extend upstream's config lists for fp8 and re-benchmark fp8 vs bf16 *within the patched upstream kernel* on real hardware; add the backend-side diff (or scope the RFC explicitly as kernel-only); and re-state correctness as 0.999996 only.

The one-sentence summary: the *decode technique* is real and well-engineered, but every headline **performance** number and the AC1/AC2 ✅ statuses rest on a scratch kernel with an invalid softmax reduction, an autotune diagnosis that is false on the pinned Triton version, and an engine that was never booted — and the shipped patch doesn't even contain the fix the documents describe.
