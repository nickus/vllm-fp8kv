# Upstream package

Two independent contributions, filed separately.

---

## 1. Bug report: sparse-MLA autotune key is blind to the KV dtype

**File against:** PR #47629 (`triton_mla_sparse_kernel.py`), or main once merged.

`_sparse_mla_kernel_final` and `_sparse_mla_kernel_split` are autotuned with

```python
@triton.autotune(configs=_FINAL_AUTOTUNE_CONFIGS,
                 key=["index_topk", "kv_group_num"])
```

The key does not include the KV cache dtype. Any second dtype whose load
pattern differs from bf16 therefore reuses bf16's cached config. Adding a
`fp8_ds_mla` path to this kernel, we measured on RTX 3090 (topk=2048, bs=32):

| fp8 config source | time |
|---|---|
| inherited from bf16 (BLOCK_N=16) | 608 µs |
| its own autotuned config (BLOCK_N=64, 8 warps) | **201 µs** |

**3.0×.** This is latent today (only bf16 exists on this path) but will bite
the moment a second dtype lands — which #47629's own SM120 discussion
anticipates. Fix is one line: add the dtype flag to the autotune `key`.

## 2. Feature: fp8 KV cache on sm_80 / sm_86 (`fp8_ds_mla`)

**Patch:** `patches/triton_mla_sparse_fp8.patch` (~230 lines, one flag, one
branch, one dispatcher arg). Generated by `patches/make_fp8_kernel_patch.py`
so it can be re-derived against a moving upstream.

### The gap, in upstream's own words

`TRITON_MLA_SPARSE` declares `supported_kv_cache_dtypes = ["auto", "float16",
"bfloat16"]`, its decode entry point is literally `_forward_bf16_kv`, and
`forward_mqa` opens with:

```python
if is_quantized_kv_cache(self.kv_cache_dtype):
    raise NotImplementedError("FP8 kv is not supported with XPU MLA Sparse yet")
```

The stated reason for excluding fp8 on Ampere is that *"Triton fp8e4nv store …
does not compile on SM80"*. Meanwhile `FLASHINFER_MLA_SPARSE_SM120` **requires**
fp8 KV — so the sparse-MLA path already wants fp8; only the Ampere kernel
support is missing. On 24 GB cards this is the binding constraint: bf16 costs
1152 B/token vs fp8_ds_mla's 656 B — **1.76× of KV capacity left on the table**
(a 2026-07-07 field report on #47629 runs GLM-5.2 on a fleet including 8× RTX
3090 and hits exactly this ceiling).

### The proposal: fp8-as-storage ≠ native fp8 compute

We never ask Triton to convert `fp8e4nv`. The standard `fp8_ds_mla` pages are
loaded as raw `uint8` and decoded in-register with bit-math that is **bit-exact
against `torch.float8_e4m3fn` over all 256 byte values**. Nothing about the
cache layout changes; the existing C++ `concat_and_cache_ds_mla` writer already
runs on sm_86 unmodified (`ENABLE_FP8` is CUDA-version-gated, not arch-gated).

This distinction matters because merged PR #43914 and open PR #47060 enforce
"native FP8 requires SM89+" — correctly, for native conversion. Software
dequant is a different mechanism and is not covered by that constraint.

**Precedent already in-tree and accepted:**
`vllm/models/deepseek_v4/xpu/xpu_sparse_decode_fp8.py` —

> *"Strategy: dequantize FP8 UE8M0 KV cache pages to BF16 on the fly, then reuse
> the BF16 sparse MLA attention kernel … This keeps the external KV cache layout
> identical to CUDA/ROCm."*

Identical architecture; we bring it to CUDA sm_8x.

### Measured (RTX 3090, sm_86)

| | |
|---|---|
| Correctness vs fp32 reference | **cosine 0.999873 – 0.999996** |
| Over pages written by vLLM's own C++ writer | **cosine 0.999996** |
| Through `triton_convert_req_index_to_global_index` + the real backend | **cosine 0.999996** |
| KV pool | **1.756× bf16** (26.2 M vs 14.9 M tokens / 16 GiB) |
| Decode speed vs bf16 (same kernel, per-dtype autotune) | **0.85 – 1.02×** |
| Non-Ampere regression | none — `IS_FP8` defaults False; bf16 path unchanged |

### Notes for reviewers

* `fp8_ds_mla` carries its scales **inline, per 128-element tile**; the layer's
  `_k_scale` is threaded into the writer but that kernel ignores it. Our decode
  therefore applies only the inline tile scales. (Verified with a tripwire: we
  pass `k_scale=2.0` into the writer and confirm the round-trip is unaffected.)
* The RoPE half of the row is **raw bf16, never quantized** — it must not be
  scaled.
* Slot offsets are computed in **int64**: `slot × 656` exceeds int32 past ~3.3 M
  slots, which a 24 GB fp8 pool reaches (26 M tokens / 16 GiB).
* Contribution #1 above (the autotune key) is a **prerequisite** for this
  feature performing correctly.

### Provenance
